## 第 1 轮：数组形式 `jmid` 绕过统一错误响应

状态：已修复并验证。

根因证据：入口将 `$_GET['jmid']` 直接传给声明为 `string` 的 `InputValidator::parseJmId()`；请求 `?jmid[]=1` 会产生 PHP 数组，并在主执行 `try` 之外触发未捕获 `TypeError`。

修复前复现：HTTP 500、`Content-Type: text/html`、空响应体；PHP 8.3.32 日志为 `InputValidator::parseJmId(): Argument #1 ($raw) must be of type string, array given`。

RED：`tests/input-validation-runtime.ps1` 要求该输入返回统一的 HTTP 400 JSON。

修复：将 `InputValidator::parseJmId()` 的入口类型改为 `mixed`，在任何字符串处理之前拒绝非标量值，再显式转换标量。这样数组输入进入既有 `SecurityException` 路径，不再触发未捕获类型错误。

GREEN：

- `tests/input-validation-runtime.ps1`：通过；数组 `jmid` 返回 HTTP 400、`application/json` 和标准失败载荷。
- `php -l index.php`：通过。

结论：这是实际 bug，不是误报。
