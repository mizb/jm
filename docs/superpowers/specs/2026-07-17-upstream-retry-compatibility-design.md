# 上游重试兼容性修复设计

日期：2026-07-17

## 问题与证据

同一部署环境中，`D:\jm\jm-boom-master\jmcomic-api-main` 老版可正常加载，而当前 API 返回 HTTP 502。已完成以下排除：

- 扩展的 `list=weekly` 映射正确，实际失败端点是上游 `/week`。
- 最新域名配置解密后仍是代码内相同的五个域名，域名刷新不是新旧差异。
- 五域中至少三个当前可直接返回 HTTP 200，且没有 HTTP 重定向；禁用自动重定向不是本次根因。
- 老版使用“每域最多 3 次、重试间隔 300ms、总计最多 15 次”；新版 `.2` 快速轮询五域两轮，默认最多 10 次。本机同一 PHP 对照中，新版 10 次在约 2.7 秒全部遇到 TLS reset，老版会继续到 15 次并持续约 7.2 秒。用户生产环境已证明老策略可成功。

因此根因是性能优化收紧了瞬时 TLS/连接失败的恢复窗口，形成可用性回归。

## 方案比较

1. **恢复老版每域三次重试，同时保留新版总预算（采用）**：与已知可用行为一致；12 秒 wall budget 防止老版最坏情况下的长尾；错误分类、安全校验、缓存与诊断保持不变。
2. 只把轮询次数从 10 提到 15：能扩大机会，但仍连续快速冲击不同域名，没有恢复老版同域短暂抖动后的恢复窗口。
3. 首请求同步刷新域名或重新允许自动重定向：当前域名配置与静态列表相同，且成功响应没有重定向，无法针对已证实差异。

## 目标行为

- API 最终版本提升为 `2026.07.17.4`（`.3` 先恢复重试窗口，`.4` 再修复真实 weekly type ID 契约）。
- 默认 `JM_MAX_UPSTREAM_ATTEMPTS=15`。
- `callJson()` 与 `fetchScrambleId()` 都按健康排序后的域名逐一处理；每个域最多三次。
- 同一域的第 2、3 次生产重试前等待 300ms；测试模式不实际等待，但验证调用顺序。
- 每次真实请求仍先取得共享预算许可；12 秒 wall budget 或 15 次 attempt budget 任一耗尽即停止。
- 仅网络错误及可重试 HTTP 状态进入重试；JSON、响应结构、解密、业务错误继续立即失败且不切域。
- HTTP 429 继续优先遵守受剩余总预算约束的 `Retry-After`；没有有效等待值时使用 300ms 短间隔。

## 验证

- 先添加回归用例：主域前两次 TLS 失败、第三次成功，断言三次都访问主域；当前实现应先失败。
- 修复后仅运行上游策略运行测试、相关合同、PHP lint 和两项目受影响的版本合同。
- 不重跑性能矩阵、APK 构建或与本次网络策略无关的测试。

## 边界

本机到五个上游域名的 PHP cURL 均被网络设备 reset，因此无法把本机公网失败冒充用户服务器结果。最终现场验收必须部署 `.4` 后，以 API 日志/响应头中的版本、attempt 数及 Suwayomi 实际加载结果为准。

## 现场后续：weekly type ID 兼容

用户部署后仍在约 1 秒内收到 502，容器日志只有 `request failure ... JmException code=502`，没有任何网络失败记录。使用相同 token 读取并解密真实 `/week` 后确认：`categories[0].id="249"`，但 `type` 的三个合法 ID 是 `hanman / another / manga`。当前代码错误地把 type ID 与 category ID 一样限制为纯数字，因此在成功的上游 200 后抛出 `Weekly defaults unavailable`。

`.4` 保持 category ID 为 1～20 位数字；type ID 允许旧数字格式或 1～32 位安全 ASCII slug（首字符字母，后续仅字母、数字、下划线、连字符）。同一规则用于生产 payload、缓存条目和外部 `type_id`，而 `category_id` 仍拒绝 slug。路径分隔符、空白和超长值继续失败。针对性完整链路 fixture 使用 `type=hanman`，已验证 `/week → 缓存校验 → /week/filter` 返回 HTTP 200。
